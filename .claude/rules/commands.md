# Commands

- `just dev` — Run backend (axum on :3002) + frontend (Vite on :5273, proxies `/api` → :3002) together for development. Polls the backend's `/api/health` before starting Vite so the SPA's first `/api/auth/me` doesn't race the (slower) axum boot and log `ECONNREFUSED` proxy errors; it falls through to starting Vite anyway if the backend exits early or after a ~120s cap (kammerz-u07)
- `just dev-backend` / `just dev-frontend` — Run either half alone
- `just build` — Production build: `frontend/build` (Vite) then `cargo build --release` (embeds it). Binary at `target/release/kammerz`
- `just fmt` — Format everything: `dprint fmt` (Markdown/JSON/TOML/YAML, config in `dprint.jsonc`) + Prettier via `bun run format` in `frontend/` (Svelte/TS/CSS, `prettier-plugin-svelte`, config in `frontend/.prettierrc`) + `cargo fmt --all`. **Run before committing.** `just fmt-check` is the read-only variant.
- rustfmt on a single root file follows its `mod` tree (`rustfmt src/main.rs` reformats every declared module) — use `just fmt`, or revert collateral files before committing.
- `just check` — Delegates to `fmt-check` + `ci-backend` + `ci-frontend` (formatting, cargo build/clippy/test `--locked`, frozen bun install, svelte-check, frontend build). All are hard gates. **Run this and ensure it passes before opening a PR.**
- **Clippy is a hard gate** (kammerz-062). `ci-backend` runs `cargo clippy --workspace --all-targets --locked -- -D warnings` between the `build` and `test` steps — so every lint fails the build, and the gate is reachable from `just check`, `just ci`, AND the GitHub `backend` job (which calls `ci-backend` and adds `components: clippy` to its toolchain step). The justfile recipe is the single source of truth for the command; ci.yml only supplies the clippy component. Prefer fixing a lint over `#[allow(...)]`; allow only with a justifying comment for a genuine false positive.
- `cargo test -p kammerz` — Backend integration tests (in-memory SQLite, real migrations + seed)
- **Frontend unit tests** — **vitest** (`frontend/vitest.config.ts`, node env, `$lib` alias). Co-located as `src/lib/utils/*.test.ts`. Run with `cd frontend && bun run test:unit` (one-shot, `vitest run`) or `bun run test:unit:watch` (watch). **They run as a gate inside `ci-frontend`** (so logic tests don't rot), mirroring how backend tests gate in `ci-backend`. vitest's `include` is scoped to `src/**` so it never tries to run the Playwright e2e specs under `frontend/tests/` (whose default glob it would otherwise match). Today only the pure-logic utils have unit coverage; components/routes are still e2e-only.
- `just coverage-frontend` — Frontend unit-test coverage (vitest + v8), scoped to `src/lib/utils`. Per-file table + HTML report at `frontend/coverage/index.html` (gitignored). **Local report only** — the tests themselves gate in `ci-frontend`; this is just the coverage view. `focus.ts`/`shot-entry.ts` read 0% by design (DOM action / API wrapper — different harness, not in the first batch).
- `just coverage` — Backend test coverage via **cargo-llvm-cov**. Prints a per-file + total summary table to stdout and writes a browseable HTML report to `target/llvm-cov/html/index.html` (under the already-ignored `/target`, so nothing leaks into git). `just coverage-open` does the same and opens the report. **Local report only — not a CI gate or a `just check` step.** One-time setup: `cargo install cargo-llvm-cov` + `rustup component add llvm-tools-preview`. The instrumented build lives in its own `target/llvm-cov-target` dir, so the first run recompiles the workspace and won't reuse the normal cache. To print a fresh summary without re-running tests after a run, use `cargo llvm-cov report --summary-only`; narrow to the app crate with `cargo llvm-cov -p kammerz`. Note `main.rs` reads ~0% (binary entrypoint — its testable logic is extracted into lib modules like `compression`/`spa` that the `tests/` suite covers).
- **CI** (`.github/workflows/ci.yml`) runs on every PR and push to `main`: a `format` job (`just fmt-check`), a `backend` job (`just ci-backend`), a `frontend` job (`just ci-frontend`), and an `e2e` job (`just e2e`, Playwright `smoke.spec.ts` against the release binary on :3002). Each job installs `just` (`taiki-e/install-action`) and calls the matching recipe — **the justfile is the single source of truth for what each job runs**; ci.yml only owns Actions-specific concerns (checkout/toolchain setup, caching, the `BUN_VERSION` pin, artifact upload). `format`/`backend`/`frontend` are unconditional; mirror them locally with `just check`.
  - **`all-checks` aggregation job** (`needs: [changes, format, backend, frontend, e2e]`, `if: always()`): the single status to require in branch protection. It fails if any dependency `result` is `failure`/`cancelled` and treats `skipped` as acceptable — so a deliberately-skipped `e2e` (docs/beads-only PR) stays mergeable, while a broken `changes` filter job still blocks. Requiring only `all-checks` lets per-job names be renamed/conditioned without editing the protection rule (the manual GitHub-settings step to switch the required check to `all-checks` lives in `workflow.md`).
  - **`e2e` is path-filtered** via a `changes` job (`dorny/paths-filter@v3`, scoped `pull-requests: read`). On push-to-main e2e always runs (full post-merge gate) via an `always() && (...)` guard — the `always()` drops GitHub's implicit needs-success gate so the boolean is the sole gate, and the dorny **filter step runs on PRs only** (skipped on push), sidestepping dorny's force-push/first-push/shallow-checkout failure modes. On a PR e2e runs only when the diff touches code/build inputs (`src/**`, `frontend/src/**`, `frontend/static/**`, `entity/**`, `migration/**`, `Cargo.{toml,lock}`, `build.rs`, `frontend/{package.json,bun.lock,vite.config.ts,svelte.config.js,playwright.config.ts}`, `frontend/tests/**`, `justfile`, `ci.yml`). A `.beads/**`- or `**/*.md`-only PR skips e2e. The `changes` **job** still runs on both events so `e2e`'s `needs: changes` never trips the skipped-dependency gotcha.
  - **Playwright browser is cached** (`actions/cache@v4` on `~/.cache/ms-playwright`, keyed `${{ runner.os }}-playwright-${{ hashFiles('frontend/bun.lock') }}`) before the `just e2e` step — `bunx playwright install` no-ops on a hit instead of re-downloading Chromium. Only the binary is cached; the `--with-deps` apt libs re-install every run.
- `just ci` — Full local mirror of the GitHub Actions pipeline, all jobs in order: `fmt-check`, `ci-backend` (`cargo build`/`clippy`/`test --locked`), `ci-frontend` (frozen `bun install` + check + build), `e2e` (Playwright smoke against the release binary on :3002, DB under /tmp). **GitHub Actions CI is active again (the repo is public, 2026-06-14) — the real `ci.yml` checks on the PR are the gate. Run `just ci` locally as pre-flight before pushing so you don't burn a CI round-trip on a failure you could've caught; you no longer need to post the `just ci` output as a PR comment (that was the workaround for when Actions was billing-disabled).** It warns (without failing) on a dirty tree and on a local bun version that differs from ci.yml's pinned `BUN_VERSION`.
- `just bootstrap <user>@<host>` — **First-time provisioning, run ONCE before the first `just deploy` on a fresh box.** `deploy` is a steady-state redeploy that assumes the box is already set up — on an unprovisioned box its first `tee /opt/kammerz/kammerz.new` fails with `No such file or directory` (the dir/user don't exist). `bootstrap` is idempotent: creates the `kammerz` system user + `/opt/kammerz/data`, installs `deploy/.env` → `/opt/kammerz/.env` (mode 600, owner kammerz, **never** clobbering an existing one), and installs+enables the systemd unit. Prereq: copy `deploy/.env.example` → `deploy/.env` (gitignored) and set `KAMMERZ_PASSWORD_HASH` first. Two `.env` landmines the example documents: `DATABASE_URL` must point into `/opt/kammerz/data` (the only `ProtectSystem=strict`-writable path, else the service crash-loops), and `SECURE_COOKIES` must stay false over plain http (LAN/VPN) or login silently fails. Mirrors the `setup-remote` pattern in the sibling `chorez`/`fewd` projects. Requires passwordless sudo on the box (same as deploy).
- `just deploy <user>@<host> [port]` — Release = deploy: runs `ci-backend` (cargo test), cross-compiles for the aarch64 DietPi server (`just build-linux`; messense gcc toolchain + rustup target, linker set in the recipe env), uploads the binary next to the live one and swaps atomically, installs the systemd unit, restarts, and polls `GET /api/health` until it reports the locally-built **git SHA** (embedded by `build.rs` as `KAMMERZ_BUILD_SHA`; also in the startup log). Requires passwordless sudo on the box. There are no GitHub releases.
- **Deployed app is reached at `http://kammerz.local`** (HTTP only — no CA signs `.local`). Set up **manually on the box** (kammerz's `just deploy` does NOT manage the proxy/alias, unlike the sibling `chorez`/`fewd` deploys which auto-sync them): a **Caddy** block in `/etc/caddy/Caddyfile` (the box's single source of truth for hostnames) — `http://kammerz.local { reverse_proxy localhost:3002 }` — plus `sudo systemctl enable --now avahi-alias@kammerz.local.service` for the mDNS name. To change/remove it, edit the box's Caddyfile (`sudo caddy validate --adapter caddyfile --config /etc/caddy/Caddyfile` then `sudo systemctl reload caddy`) and the avahi-alias unit directly. Box is `dietpi@dietpi.home` (192.168.10.4); DB lives at `/opt/kammerz/data/kammerz.db`.
- **Replace the deployed DB with the local catalog** (full overwrite, not a merge): `sqlite3 kammerz.db ".backup /tmp/snap.db"` (WAL-safe — never plain-`cp` a live WAL DB), then `ssh dietpi@dietpi.home 'cat > /tmp/snap.db' < /tmp/snap.db` (**`scp` fails** — the box's fish login shell breaks scp's sftp subsystem, so use `ssh … cat`, same reason `just deploy` pipes via `tee`). Then on the box: `sudo systemctl stop kammerz`, `sudo rm -f /opt/kammerz/data/kammerz.db-{wal,shm}` (else the old WAL replays onto the new DB → corruption), `sudo mv /tmp/snap.db /opt/kammerz/data/kammerz.db`, `sudo chown kammerz:kammerz …/kammerz.db`, `sudo systemctl start kammerz`. Migrations are a no-op (`seaql_migrations` already populated); the app also self-backs-up as `kammerz.db.pre-migrate-*` on each migrating start.
- `echo -n <pw> | kammerz hash-password` — Generate the argon2 hash for `KAMMERZ_PASSWORD_HASH`. **Reads the password from stdin, never argv** (argv leaks into shell history / `ps`). On a TTY it prompts with echo off.
- **Build wipes `frontend/build/`** — `bun run build` (and thus `just check` / `just build`) deletes the tracked `frontend/build/.gitkeep` (the adapter-static output dir is regenerated). Restore it before committing: `git checkout -- frontend/build/.gitkeep`. Don't commit its deletion.
- **Verification:** This is a normal browser app — browser/Playwright verification is valid. Run via `just dev` (axum :3002 + Vite :5273 proxy) and open `http://localhost:5273`, or build and run the release binary on :3002. Verify backend with `cargo test`, frontend markup/types with `bun run build` / `bun run check`, data with `sqlite3` queries against the configured `DATABASE_URL` (dev default `./kammerz.db`).
- To manually verify the release binary **without colliding** with `just dev`/`just ci` on `:3002`, run it on an alternate port with a throwaway DB: `PORT=3199 DATABASE_URL=sqlite:/tmp/v.db?mode=rwc target/release/kammerz` (kill it after). Useful for `curl` checks of headers/compression/ETag against the real embedded build.
- **Playwright MCP `browser_take_screenshot` writes relative to the main project root**, not the cwd/worktree — the PNG lands in the repo root; pass an absolute `filename` and delete the stray file before committing.
